Integration Guide

Complete documentation for integrating @stvor/web3 — post-quantum E2EE — into Web3 apps, TON DApps, and AI agent systems. Zero external dependencies. NIST FIPS 203.

Installation

terminal bash 
# npm
npm install @stvor/web3

# yarn
yarn add @stvor/web3

# pnpm
pnpm add @stvor/web3

Requirements

Node.js ≥ 18 or any modern browser (Chrome 90+, Firefox 89+, Safari 15+). Requires WebAssembly and Web Crypto API — available everywhere since 2022.

ℹ️
The package ships a 270 KB Rust WASM binary. No native addons, no node-gyp. Works in Cloudflare Workers, Vercel Edge, Deno, Bun.

Quick Start

Two imports, three lines. The WASM module must be initialised once before use.

quickstart.ts typescript 
import { StvorWeb3 } from '@stvor/web3';
import initWasm     from '@stvor/web3/wasm';

// 1. Load WASM crypto engine (once, at app startup)
const wasm = await initWasm();

// 2. Connect — derives quantum-safe identity from your wallet/signer
const alice = await StvorWeb3.connect({ provider, wasm });

// 3. Send end-to-end encrypted message
await alice.send(bobAddress, { text: 'Hello, quantum-safe world!' });

// 4. Receive — decrypted automatically
alice.onMessage(msg => console.log(msg.from, msg.data));

// 5. Cleanup
await alice.disconnect();

Architecture

STVOR uses a hybrid post-quantum scheme: ML-KEM-768 (quantum-safe) combined with P-256 ECDH (classical). Breaking the encryption requires breaking both simultaneously — impossible with any known algorithm.

crypto stack text 
Identity:     Wallet address  (TON / EVM / any signer)
Key exchange: Hybrid X3DH
               ├── P-256 ECDH        (classical, NIST)
               └── ML-KEM-768        (post-quantum, NIST FIPS 203)
               └── HKDF-SHA256 combine → shared_secret
Sessions:     Double Ratchet  (Signal Protocol)
               ├── Sending chain ratchet  → fresh key per message
               └── DH ratchet            → break-in recovery
Encryption:   AES-256-GCM     (authenticated)
Signing:      ECDSA P-256 / SHA-256
Core:         Rust → WASM     (270 KB, zero C deps, memory-safe)
Hybrid security: shared_key = HKDF(ecdh_sk ‖ mlkem_ss, "STVOR-HYBRID-v1"). If classical ECDH is broken by a quantum computer, ML-KEM-768 still protects the key. If ML-KEM has a future flaw, ECDH still holds.
Web3 · EVM

Wallet Identity (EVM)

Any EIP-1193 wallet — MetaMask, Coinbase, Ledger, Rainbow — becomes a cryptographic identity. The user signs a deterministic message once; STVOR derives their P-256 + ML-KEM-768 keypairs via HKDF. Same wallet → same keys, always.

1
User signs the STVOR derivation message
The message is STVOR-AA-EVM-v1:{chainId}:{address}. This signature is the entropy source — never sent anywhere.
2
HKDF derives P-256 identity keypair + signed pre-key
IK and SPK are derived deterministically from the signature. ML-KEM-768 keypair is generated fresh (randomised).
3
Public keys registered on-chain (optional)
For TON: stored in stvor_registry.fc smart contract. For EVM-only use, exchange public keys out-of-band or via your own registry.
evm-identity.ts typescript 
import { StvorAA } from '@stvor/web3';
import initWasm   from '@stvor/web3/wasm';

const wasm = await initWasm();

// Works with MetaMask, Coinbase, Ledger, WalletConnect…
const client = await StvorAA.connectEVM({
  provider: window.ethereum,   // any EIP-1193 provider
  chainId: 1,                  // Ethereum mainnet (or 137, 56, …)
  wasm,
});

console.log(client.address);   // '0xabc...' — EVM address
console.log(client.publicKey);  // P-256 identity key (base64url)

// Identity derivation is deterministic:
// personal_sign("STVOR-AA-EVM-v1:1:0xabc...")
//   → HKDF(..., "IK",  32) → P-256 identity keypair
//   → HKDF(..., "SPK", 32) → P-256 signed pre-key
//   → wasm_mlkem_keygen()   → ML-KEM-768 keypair
Web3 · EVM

E2EE Session Setup

The first time Alice and Bob communicate, a hybrid X3DH key exchange establishes a shared secret. All subsequent messages use the Double Ratchet — each message has a unique key.

evm-session.ts typescript 
import { StvorAA } from '@stvor/web3';
import initWasm   from '@stvor/web3/wasm';

const wasm = await initWasm();

// Alice connects
const alice = await StvorAA.connectEVM({ provider: aliceProvider, chainId: 1, wasm });

// Bob connects (different wallet)
const bob = await StvorAA.connectEVM({ provider: bobProvider, chainId: 1, wasm });

// Alice sends — hybrid X3DH happens automatically on first message
await alice.send(bob.address, {
  text: 'First message — session established with hybrid X3DH',
});

// Bob receives and replies — Double Ratchet advances, new key per message
bob.onMessage(async msg => {
  console.log('From:', msg.from);   // '0xalice...'
  console.log('Data:', msg.data);   // { text: 'First message...' }
  console.log('At:',   msg.timestamp); // Date

  await bob.send(msg.from, { text: 'Reply — different key than Alice's message' });
});
Web3 · EVM

Send & Receive — Full Example

Any JSON-serialisable payload is supported. The library handles serialisation, encryption, and delivery.

evm-chat.ts typescript 
import { StvorAA } from '@stvor/web3';
import initWasm   from '@stvor/web3/wasm';

const wasm   = await initWasm();
const client = await StvorAA.connectEVM({ provider: window.ethereum, chainId: 1, wasm });

// ── Send any structured data ──────────────────────────────
await client.send('0xrecipient...', {
  text:          'Private governance vote',
  vote:          'yes',
  proposalId:    42,
  rationale:     'Increases treasury APY',
  confidential:  true,
});

// ── Listen for all incoming messages ─────────────────────
const unsub = client.onMessage(msg => {
  // msg.from      — sender address '0x...'
  // msg.data      — decrypted JSON payload
  // msg.timestamp — Date object
  // msg.id        — unique message ID

  if (msg.data.vote) {
    console.log(`${msg.from} voted ${msg.data.vote} on proposal ${msg.data.proposalId}`);
  }
});

// Unsubscribe when done
unsub();

// ── Disconnect (stops polling, clears session state) ──────
await client.disconnect();
Account Abstraction · ERC-4337

Account Abstraction (ERC-4337)

Safe, Coinbase Smart Wallet, Biconomy, ZeroDev — any ERC-4337 wallet. STVOR can cryptographically bind an E2EE session to a specific UserOperation, proving that the encrypted channel belongs to the AA wallet that signed the op.

aa-binding.ts typescript 
import { StvorAA } from '@stvor/web3';
import initWasm   from '@stvor/web3/wasm';

const wasm   = await initWasm();
const client = await StvorAA.connectEVM({
  provider: window.ethereum, // Safe, Coinbase, Biconomy — any EIP-1193
  chainId: 1,
  wasm,
});

// ── Bind E2EE session to a UserOperation ──────────────────
// userOpHash comes from your bundler (EIP-4337)
const binding = client.bindUserOp(userOpHash, sessionRootKey);

// binding = {
//   userOpHash:        '0xdeadbeef...'      — the op being bound
//   identitySig:       'base64url...'       — ECDSA sig over (userOpHash ‖ sessionCommitment)
//   sessionCommitment: 'base64url...'       — HKDF commitment of session root key
// }

// ── Verify peer's binding ─────────────────────────────────
const valid = client.verifyUserOp(binding, peerIdentityKey, sessionRootKey);
console.log('Binding valid:', valid); // true

// ── Send within the bound session ─────────────────────────
await client.send(peerAddress, {
  action:     'execute_swap',
  tokenIn:    'USDC',
  tokenOut:   'ETH',
  amount:     '1000',
  boundToOp:  binding.userOpHash,
});
Why bind to a UserOperation? It proves that the encrypted instructions originated from the same wallet that signed the on-chain op. An attacker who intercepts the encrypted channel cannot replay it for a different op.

On-chain Key Registry

Public keys are stored in the stvor_registry.fc FunC contract. This makes key discovery trustless — anyone can verify your identity key without a central server.

ton-registry.ts typescript 
import { TonStorageClient } from '@stvor/web3';

const registry = new TonStorageClient({
  tonApiUrl:        'https://testnet.toncenter.com/api/v2',
  contractAddress:  'EQD...',
  apiKey:           process.env.TON_API_KEY,
});

// ── Register your public keys on-chain ────────────────────
// Build the cell body (pure JS, no @ton/core needed)
const body = registry.buildRegisterKeysBody({
  identityKey:          client.publicKey.identityKey,      // P-256, 65 bytes
  signedPreKey:         client.publicKey.signedPreKey,     // P-256, 65 bytes
  signedPreKeySignature: client.publicKey.spkSig,          // ECDSA DER sig
  mlkemEncapKey:        client.publicKey.mlkemEncapKey,    // ML-KEM-768, 1184 bytes
});

// Sign via TonConnect and broadcast
const signedBoc = await tonConnect.sendTransaction({ messages: [{ address: contractAddress, amount: '50000000', payload: body }], validUntil: Date.now() / 1000 + 300 });
await registry.registerKeys(client.publicKey, signedBoc);

// ── Look up anyone's public keys ──────────────────────────
const bobKeys = await registry.fetchKeys('0:bob_address...');
// bobKeys = { identityKey, signedPreKey, signedPreKeySignature, mlkemEncapKey, address }

TON Wallet v5 Extension Signing

TON Wallet v5 supports native extensions. STVOR can sign and verify extension bodies, binding your E2EE session to a specific wallet v5 extension operation.

ton-v5-extension.ts typescript 
import { StvorAA } from '@stvor/web3';

AI Agents

Agent Identity

Each AI agent gets a unique, deterministic cryptographic identity derived from its wallet signature. The same agent key always produces the same STVOR identity — reproducible across restarts, upgrades, and migrations.

An AI agent is just a signer with a private key. It can be an EOA, a Safe multisig, a Biconomy AA wallet, or any programmatic signer. STVOR treats it identically to a human wallet.
agent-identity.ts typescript 
import { StvorAA }      from '@stvor/web3';
import initWasm          from '@stvor/web3/wasm';
import { Wallet }       from 'ethers';    // or viem, web3.js, etc.

const wasm = await initWasm();

// Agent's deterministic private key (from env / KMS / HSM)
const agentWallet = new Wallet(process.env.AGENT_PRIVATE_KEY!);

// Wrap as EIP-1193 provider
const agentProvider = {
  request: async ({ method, params }: { method: string; params?: unknown[] }) => {
    if (method === 'eth_accounts') return [agentWallet.address];
    if (method === 'eth_chainId')  return '0x1';
    if (method === 'personal_sign') {
      const [msg] = params as string[];
      return agentWallet.signMessage(Buffer.from(msg.slice(2), 'hex'));
    }
    throw new Error(`Unsupported: ${method}`);
  },
};

// Connect — identity derived from agent's wallet signature
const agent = await StvorAA.connectEVM({
  provider: agentProvider,
  chainId:  1,
  wasm,
});

console.log('Agent address:', agent.address);
// Same address → same STVOR identity, always

AI Agents

Encrypted Long-term Memory

AI agents accumulate context over time — conversation history, learned user preferences, private instructions, fine-tuning data. STVOR encrypts this memory before storing it in decentralised storage (TON Storage, IPFS, Arweave). Past context stays private forever, not just in transit.

Without encryption, an agent's Long-term Memory stored on-chain or in IPFS is publicly readable. Anyone — including competing agents, MEV bots, and future quantum computers — can read it.
agent-memory.ts typescript 
import { StvorAA }  from '@stvor/web3';
import initWasm     from '@stvor/web3/wasm';
import initWasmAPI, { WasmSession, wasm_mlkem_keygen, wasm_mlkem_encaps, wasm_mlkem_decaps } from '@stvor/web3/wasm';

const wasm  = await initWasm();
const agent = await StvorAA.connectEVM({ provider: agentProvider, chainId: 1, wasm });

// ── Encrypt memory blob before storing off-chain ──────────
async function encryptMemory(memory: object, recipientAddress: string): Promise<string> {
  // "Send" to yourself — uses your own public key as recipient
  // In practice: create a dedicated memory encryption session
  const payload = { type: 'long_term_memory', data: memory, ts: Date.now() };
  await agent.send(recipientAddress, payload);
  // Returns encrypted bag ID stored on TON Storage
  return 'bag_id...';
}

// ── Store encrypted agent context ─────────────────────────
const agentMemory = {
  userPreferences: { theme: 'dark', language: 'en', riskTolerance: 'low' },
  conversationHistory: [
    { role: 'user',  content: 'Buy TON when RSI drops below 30' },
    { role: 'agent', content: 'Understood. Monitoring RSI...' },
  ],
  learnedPatterns: [
    'User prefers TON-native assets',
    'Avoid trades after 22:00 UTC',
    'Maximum single-trade exposure: 5%',
  ],
  privateSystemPrompt: 'You are a conservative DeFi agent. Never recommend leverage above 2x.',
};

// Encrypted — only this agent (and authorised peers) can read it
await encryptMemory(agentMemory, agent.address);

// ── Retrieve and decrypt ──────────────────────────────────
agent.onMessage(msg => {
  if (msg.data.type === 'long_term_memory') {
    // Restore context — decrypted automatically
    restoreContext(msg.data.data);
  }
});

Memory Security Guarantees

PropertyGuarantee
At restAES-256-GCM encrypted before upload. The decentralised storage node sees only ciphertext.
In transitTLS + Double Ratchet. Each memory write uses a fresh message key.
Forward secrecyCompromising today's key doesn't expose past memory writes.
Quantum resistanceML-KEM-768 protects the session key — safe against future quantum decryption.

Multi-agent Systems

STVOR enables secure communication in multi-agent pipelines. Each pair of agents maintains a separate Double Ratchet session — compromising one agent doesn't expose other channels.

multi-agent.ts typescript 
import { StvorAA } from '@stvor/web3';
import initWasm   from '@stvor/web3/wasm';

const wasm = await initWasm();

// ── Spawn agent fleet ─────────────────────────────────────
const [coordinator, researcher, executor, auditor] = await Promise.all([
  StvorAA.connectEVM({ provider: coordinatorProvider, chainId: 1, wasm }),
  StvorAA.connectEVM({ provider: researcherProvider,  chainId: 1, wasm }),
  StvorAA.connectEVM({ provider: executorProvider,    chainId: 1, wasm }),
  StvorAA.connectEVM({ provider: auditorProvider,     chainId: 1, wasm }),
]);

// ── Coordinator broadcasts private mission ────────────────
const mission = {
  type:       'defi_arbitrage',
  target:     'TON/USDT',
  maxSlippage: 0.5,
  secret:     'internal strategy — not for public chains',
};

await Promise.all([
  coordinator.send(researcher.address, { ...mission, role: 'research' }),
  coordinator.send(executor.address,   { ...mission, role: 'execute'  }),
  coordinator.send(auditor.address,    { ...mission, role: 'audit'   }),
]);
// Each channel is independently encrypted — Researcher ≠ Executor session

// ── Agents report back privately ──────────────────────────
researcher.onMessage(async msg => {
  const analysis = await doResearch(msg.data);
  await researcher.send(coordinator.address, { type: 'research_done', analysis });
});

executor.onMessage(async msg => {
  const txHash = await executeStrategy(msg.data);
  await executor.send(auditor.address, { type: 'execution_report', txHash });
});

Low-level WASM API

ML-KEM-768 Primitives

Direct access to the Rust crypto core via WASM bindings. Use when you need fine-grained control over the key encapsulation mechanism.

mlkem-primitives.ts typescript 
import initWasm, {
  wasm_mlkem_keygen,
  wasm_mlkem_encaps,
  wasm_mlkem_decaps,
} from '@stvor/web3/wasm';

await initWasm();

// ── Key generation ────────────────────────────────────────
const { ek, dk } = JSON.parse(wasm_mlkem_keygen());
// ek — 1184-byte encapsulation key  (base64url) — public, share freely
// dk —   64-byte decapsulation seed (base64url) — PRIVATE, keep secret
//        64 bytes = seed (d ‖ z) per FIPS 203 §7.1
//        Full 2400-byte key is re-derived from seed on each decaps call

// ── Encapsulate (sender side) ─────────────────────────────
const { ct, ss } = JSON.parse(wasm_mlkem_encaps(ek));
// ct — 1088-byte ciphertext (base64url) — send to recipient
// ss —   32-byte shared secret (base64url) — use as encryption key

// ── Decapsulate (recipient side) ──────────────────────────
const ss2 = wasm_mlkem_decaps(dk, ct);
// ss2 === ss  ✓  — same shared secret, different sides

console.log('Shared secrets match:', ss === ss2);
The dk value (64-byte seed) is your private key. Never log it, send it over the network, or store it unencrypted. The seed is sufficient to re-derive the full 2400-byte decapsulation key at any time.

Hybrid X3DH Sessions

Manual session establishment using the hybrid X3DH + Double Ratchet primitives.

hybrid-session.ts typescript 
import initWasm, {
  WasmKeyPair, WasmSession,
  wasm_mlkem_keygen,
  wasm_hybrid_session_initiate,
  wasm_hybrid_session_respond,
} from '@stvor/web3/wasm';

await initWasm();

// ── Both parties generate their keypairs ──────────────────
const aliceIK  = new WasmKeyPair();  // Identity key (P-256)
const aliceSPK = new WasmKeyPair();  // Signed pre-key (P-256)
const bobIK    = new WasmKeyPair();
const bobSPK   = new WasmKeyPair();
const bobPqc   = JSON.parse(wasm_mlkem_keygen()); // ML-KEM-768 keypair

// ── Alice initiates hybrid X3DH ───────────────────────────
const { session_json, mlkem_ct } = JSON.parse(
  wasm_hybrid_session_initiate(
    aliceIK, aliceSPK,
    bobIK.public_key, bobSPK.public_key, bobPqc.ek,
  )
);
const aliceSession = WasmSession.from_json(session_json);

// Alice sends mlkem_ct to Bob (out-of-band or in first message header)

// ── Bob responds, derives same shared secret ──────────────
const bobSession = wasm_hybrid_session_respond(
  bobIK, bobSPK,
  aliceIK.public_key, aliceSPK.public_key,
  bobPqc.dk, mlkem_ct,
);

// ── Encrypted messaging ───────────────────────────────────
const ciphertext = aliceSession.encrypt(new TextEncoder().encode('secret payload'));
const plaintext  = bobSession.decrypt(ciphertext);

console.log(new TextDecoder().decode(plaintext)); // 'secret payload'

// ── Persist session state ─────────────────────────────────
const serialised = aliceSession.to_json();   // store in DB / KV / encrypted memory
const restored   = WasmSession.from_json(serialised); // restore next startup
